# Detailed Exploration of JAXB Alternatives

This document provides a detailed evaluation of open-source alternatives to the current JAXB 2.3.1 implementation used for XML serialization in the Alpheus AFP Parser.

## 1. Current State
The project currently uses `javax.xml.bind:jaxb-api:2.3.1` and `org.glassfish.jaxb:jaxb-runtime:2.3.1`.
- **Scope**: ~46 classes annotated with `@XmlRootElement`, `@XmlElement`, etc.
- **Mechanism**: Reflection-based data binding with `Marshaller` pooling and `JAXBContext` caching.
- **Risk**: The `javax.*` namespace is effectively legacy. Modern Java development (Jakarta EE 9+) has migrated to the `jakarta.*` namespace.

---

## 2. Evaluated Alternatives

### A. Jakarta XML Binding (Jakarta EE 10/11)
The official successor to JAXB, maintained by the Eclipse Foundation.

*   **License**: Eclipse Public License 2.0 (EPL-2.0) - **100% Open Source**.
*   **Future Proofing**: Highest. This is the "standard" path for Java. It ensures compatibility with future JDKs and Jakarta EE environments.
*   **Performance**: Comparable to current JAXB (since it's an evolution of the same codebase).
*   **Implementation Effort**: Low. Requires changing package imports from `javax.xml.bind` to `jakarta.xml.bind`.
*   **Risks**: Minimal. The main "risk" is the namespace migration, which is a one-time effort.

### B. Jackson XML (jackson-dataformat-xml)
A high-performance data-binding library widely used for JSON, with an XML extension.

*   **License**: Apache License 2.0 - **100% Open Source**.
*   **Future Proofing**: High. Jackson is the de-facto standard for data binding in the Java ecosystem (Spring, Quarkus, etc.).
*   **Performance**: Higher than JAXB. Optimized for streaming and lower memory overhead.
*   **Implementation Effort**: Medium. While it supports JAXB annotations via `jackson-module-jaxb-annotations`, some XML-specific nuances (like wrapped vs. unwrapped lists) might require adjustment.
*   **Risks**: Jackson's XML support is not a 1:1 JAXB clone. As noted in their documentation, certain complex JAXB constructs might not be supported, or might behave differently.

### C. Manual StAX + Aalto XML
The lowest-level approach, using the Streaming API for XML (StAX) with the ultra-fast Aalto implementation.

*   **License**: Apache License 2.0 - **100% Open Source**.
*   **Future Proofing**: Medium-High. StAX is a stable JDK standard. Aalto is actively maintained by FasterXML.
*   **Performance**: Extreme. Bypasses all reflection and generic data-binding overhead.
*   **Implementation Effort**: High. Requires writing manual serialization logic for all 46+ structured field classes.
*   **Risks**: Maintenance burden. Any change to the AFP object model requires manual updates to the XML writer logic.

## 4. Recommendation

### Primary Recommendation: Jakarta XML Binding 4.0
For the Alpheus AFP Parser, **Jakarta XML Binding 4.0** is the recommended "future-proof" path.
- It preserves the current annotation-driven model (which is clean and declarative).
- It moves the project to the modern `jakarta.*` namespace, ensuring longevity.
- It remains 100% compatible with the existing XML schemas generated by the parser.

### Secondary Recommendation: Jackson XML (For Performance)
If the benchmarked speed of JAXB remains a bottleneck even after the Phase 1-6 optimizations, **Jackson XML** should be the next choice. It offers a significant performance boost without the maintenance nightmare of manual StAX writing.

### Avoid: Manual StAX
Manual StAX writing should be reserved only for specific, high-frequency fields (like `NOP` or `TLE` in massive volumes) if they prove to be extreme outliers in performance profiles.
