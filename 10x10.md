# Performance Benchmark: 10x 10KB TRN-heavy AFP Files

This benchmark evaluates the performance of the AFP-to-XML conversion across 5 equally distributed releases, using a test set of 10 AFP files, each approximately 10KB, consisting primarily of Transparent Data (TRN) control sequences.

## Benchmark Environment
- **Hardware**: Sandbox Environment
- **Java Version**: 21
- **Test Data**: 10 files, ~10KB each, heavy TRN usage.
- **Methodology**: Total time for 10 full iterations (converting all 10 files 10 times).

## Results

| Release | Optimization Flags | Total Time (10 runs of 10 files) | Avg Time per run |
| :--- | :--- | :--- | :--- |
| **v1.0** | (None supported) | 48,786 ms | 4,878.6 ms |
| **v3.0** | (None supported) | 50,799 ms | 5,079.9 ms |
| **v5.0** | `-a -c` | 836 ms | 83.6 ms |
| **v7.0** | `-a -c` | 1,375 ms | 137.5 ms |
| **v9.11** | `-P -a -c` | 1,457 ms | 145.7 ms |

## Observations

1.  **Massive Performance Leap (v5.0)**: The introduction of aggressive I/O (`-a`) and optimized charset decoding (`-c`) in the v5.x era resulted in a ~60x performance improvement compared to earlier versions.
2.  **Overhead in Later Versions**: There is a slight increase in execution time from v7.0 to v9.11. This is likely attributed to the increased complexity of the parser, more comprehensive validation, and the transition to the Jackson-based `AfpJacksonXmlWriter` as the primary writer.
3.  **TRN Handling**: For TRN-heavy files of this small size (10KB), the overhead of thread orchestration in parallel mode (`-P`) in v9.11 might outweigh the benefits of multi-core processing, though it provides significant gains for larger files (as seen in other benchmarks).
4.  **Stability**: The newer versions show much more consistent performance across runs compared to the early JAXB-based versions.

## Extended Benchmark around v5.0 Optimum

This additional benchmark focuses on the releases surrounding the v5.x era to pinpoint the performance peak and evaluate the impact of the `-p` flag (where available).

| Release | Optimization Flags | Total Time (10 runs of 10 files) | Avg Time per run |
| :--- | :--- | :--- | :--- |
| **v4.3** | (None supported) | 954 ms | 95.4 ms |
| **v5.0** | (None supported) | 869 ms | 86.9 ms |
| **v5.1** | (None supported) | 800 ms | 80.0 ms |
| **v5.2** | (None supported) | 799 ms | 79.9 ms |
| **v5.3** | (None supported) | 902 ms | 90.2 ms |
| **v7.0** | `-p` (Parallel) | 813 ms | 81.3 ms |
| **v9.11** | `-p` (PTX Debug) | 867 ms | 86.7 ms |

**Notes:**
1. In version **v7.0**, the `-p` flag refers to **Parallel** processing mode.
2. Starting from version **v9.0**, the `-p` flag was reassigned to **PTX Debug** (performance monitoring for PTOCA), while Parallel mode was moved to `-P`.
3. The absolute optimum for this specific 10x 10KB TRN-heavy workload was observed at **v5.2**.
